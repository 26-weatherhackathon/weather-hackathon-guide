# 개발 실행 순서 — Vercel 배포 · Claude Pro 절약형 (Sonnet 분업 계획)

> 기상·기후 AI 해커톤 2026 출품작 · **Claude Pro 한도를 아끼며 Sonnet으로 쪼개 개발하는 순서**
> 전제: 모든 설계는 이미 문서화됨(`forecaster-build-spec.md` / `-ux-spec.md` / `-data-utilization.md`).
> 이 문서는 "어떤 순서로, 어떤 세션 단위로, 어떻게 토큰을 아끼며" 만드느냐만 다룬다.

## 0. Pro 절약 핵심 원칙 (먼저 읽기)

Claude Pro는 사용량 한도(시간/주 단위)가 있다. 한도를 태우는 건 **모델 등급 × 컨텍스트 크기 × 대화 길이**다.
아래 원칙이 이 계획 전체를 관통한다.

| 원칙 | 이유 | 실행 |
|---|---|---|
| **Sonnet으로 구현** | Opus는 한도 소모가 큼 | 빌드 세션은 `/model sonnet`. Opus는 설계·난해한 버그 1회성만 |
| **세션을 작게 쪼갠다** | 긴 대화 = 매 턴 누적 토큰 폭증 | 1세션 = 1산출물. 끝나면 `/clear`로 컨텍스트 비움 |
| **명세서가 기억이다** | 매번 재설명하면 토큰 낭비 | "build-spec §6 보고 scoring.ts 만들어" 식으로 **파일+섹션만 지목** |
| **순수 로직 먼저** | UI·이미지는 디버깅 루프가 길어 비쌈 | 타입·채점·상태는 테스트로 검증(싼 검증) 후 UI |
| **정적 데이터** | 런타임 API 호출 0 → 디버깅 루프 0 | 데이터 수집은 코딩 세션 밖에서 1회 |
| **세션마다 커밋** | 새 세션이 깨끗한 git에서 시작 | 산출물 나오면 즉시 commit+push, 다음 세션은 clean |
| **Claude 밖에서 할 일은 밖에서** | 키 발급·이미지 생성은 토큰 아까움 | API 신청·이미지 생성·Vercel 클릭은 사람이 직접 |

> **한 줄 요약**: *작은 세션 · Sonnet · 명세서 지목 · 끝나면 clear · 바로 커밋.*

## 1. 세션 단위로 쪼갠 전체 순서

각 "세션"은 **새 대화 1개**(끝나면 clear). 입력은 "어떤 명세 섹션을 읽힐지", 산출물은 "무엇이 나오는지", 🚀는 Vercel 배포 체크포인트.
budget S(짧음)/M/L = 대략적 대화 길이(=한도 소모).

### Stage A — 파이프라인부터 세운다 (배포 먼저)
| # | 세션 | 입력(읽힐 것) | 산출물 | 모델 | budget | 🚀 |
|---|---|---|---|---|---|---|
| A1 | 프로젝트 스캐폴드 | build-spec §2,§3 | Astro+Tailwind+Preact+Vercel 초기화, 빈 index | Sonnet | S | 🚀 |
| A2 | Vercel 연결·배포 | — (사람이 대시보드 클릭) | "hello" 배포 성공, 도메인 | 사람 | S | 🚀 |
| A3 | 디자인 토큰 주입 | ux-spec §1 | tailwind.config 토큰·폰트·BaseLayout | Sonnet | S | |

> **왜 배포부터?** 파이프라인이 살아있으면 이후 모든 세션이 "푸시→자동배포"로 즉시 확인됨. 막판 배포 사고 방지.

### Stage B — 순수 로직 (싸고 안전한 토대)
| # | 세션 | 입력 | 산출물 | 모델 | budget | 🚀 |
|---|---|---|---|---|---|---|
| B1 | 타입 정의 | build-spec §5 | `lib/types.ts` 전체 | Sonnet | S | |
| B2 | 채점 엔진 + 테스트 | build-spec §6 | `lib/scoring.ts` + `scoring.test.ts`(vitest 통과) | Sonnet | M | |
| B3 | 세션 저장 | build-spec §7 | `lib/session.ts`(localStorage) + 등급 매핑 | Sonnet | S | |

> 순수 함수라 **테스트로 검증**(브라우저 디버깅 불필요) → 토큰 거의 안 쓰고 끝남. 여기서 만든 게 이후 UI의 뼈대.

### Stage C — 한 케이스 골격 (가짜 데이터로 끝까지)
| # | 세션 | 입력 | 산출물 | 모델 | budget | 🚀 |
|---|---|---|---|---|---|---|
| C1 | 케이스 페이지 상태머신 | build-spec §3 트리, ux-spec §2,§3 | `case/[caseId].astro` 5단계 전환(플레이스홀더) | Sonnet | M | 🚀 |
| C2 | LayerViewer + TimeSlider | ux-spec §5.3, data-util §3 I1 | 레이어 토글·시간 스크럽(더미 이미지) | Sonnet | M | |
| C3 | ModelCompare | ux-spec §5.4 | 두 발표 비교·불일치 하이라이트(더미) | Sonnet | M | |
| C4 | ForecastInput + Evidence | ux-spec §5.5 | 슬라이더·근거체크·제출 게이팅 | Sonnet | M | |
| C5 | ResultReveal + Reflection | ux-spec §5.6, build-spec §5-C C1 | 실황공개·채점·연출(B2 채점 연결) | Sonnet | M | 🚀 |

> C 끝나면 **가짜 데이터로 전체 루프가 도는 데모**가 배포돼 있음. 여기까지가 "작동 증명". 이후는 진짜 데이터·폴리시.

### Stage D — 진짜 데이터 (코딩 세션 밖 + 연결)
| # | 세션 | 입력 | 산출물 | 모델 | budget | 🚀 |
|---|---|---|---|---|---|---|
| D0 | (사람) API 키 발급 | data-util §4 | authKey·serviceKey | 사람 | — | |
| D1 | 수집 스크립트 작성 | build-spec §4, data-util §5 | `scripts/fetch-case.ts` 등 | Sonnet | M | |
| D2 | (사람) 스크립트 실행·검증 | data-util §5 우선순위 | case2 호우 실데이터 정적 파일 | 사람+Sonnet | M | |
| D3 | 실데이터 연결 | build-spec §5 스키마 | C2~C5가 case2 meta.json 소비 | Sonnet | M | 🚀 |

> 데이터 수집(D2)은 **토큰 아까운 코딩 대신 스크립트 실행**으로. 막히는 API만 Sonnet에 짧게 물어 해결.

### Stage E — 화려함·크래프트 (집중 투자)
| # | 세션 | 입력 | 산출물 | 모델 | budget | 🚀 |

| E2 | 데이터 구동 날씨 모션 | data-util §3 I10 | 강수입자·바람결 Canvas(강수량 구동) | Sonnet | L | 🚀 |
| E3 | 레이더 타임머신+이동벡터 | data-util §3 I1,I8 | 스크럽·벡터·전레이어 동기화 | Sonnet | L | |

> E는 한도를 많이 먹는다(L). **다른 Stage 다 끝낸 뒤** 남은 한도로. E4는 시간/한도 남으면.

### Stage F — 케이스 확장 (구조 반복)
| # | 세션 | 입력 | 산출물 | 모델 | budget | 🚀 |

## 2. 권장 진행 리듬 (한도 분산)

- **하루 2~3세션**, 각 세션 끝나면 `/clear` + commit. 한도는 시간창 단위라 몰아치지 말고 분산.
- **순서 의존**: A→B→C는 직렬(앞이 뒤의 토대). D 이후는 일부 병렬 가능(데이터 수집은 사람이 백그라운드로).
- **막히면 Opus 1턴**: Sonnet이 같은 버그로 3턴 이상 헤매면, 그 한 건만 Opus로 물어보고 다시 Sonnet 복귀. (Opus 상주 금지)
- **버퍼 남기기**: 마지막 G4(배포 직전 QA)용 한도를 항상 남겨둠. E(화려함)에서 한도를 다 쓰지 말 것.

## 3. 토큰 아끼는 세션 운영 디테일

1. **세션 시작 멘트 템플릿**:
   > "build-spec §6만 읽고 `src/lib/scoring.ts`와 vitest 테스트를 만들어. 다른 파일은 읽지 마."
   → Claude가 레포 전체를 훑지 않게 **읽을 범위를 못박음**(탐색 토큰 절약).
2. **큰 파일 재읽기 금지**: 편집은 Edit로 부분만. "확인차 다시 읽기" 시키지 않기.
3. **로그·에러는 핵심만 붙여넣기**: 전체 스택트레이스 말고 관련 줄만.
4. **이미지 생성은 별도 도구**로(코딩 세션에서 생성 금지). 결과 파일만 `public/ai/`에 넣음.
5. **테스트가 있는 건 테스트로 검증**(B2). "브라우저에서 눌러봐줘" 왕복을 줄임.
6. **한 세션 = 한 PR/커밋**. 리뷰·롤백 단위가 명확해 재작업(=토큰) 감소.

## 4. Vercel 배포 운영 (사람이 1회 세팅)

```
1) GitHub 레포 → Vercel Import (Framework: Astro 자동감지)
2) Build: npm run build / Output: dist
3) 환경변수: ANTHROPIC_API_KEY (G3에서 필요, api/reflect.js 전용)
   ※ 데이터 수집 키(KMA)는 런타임 불필요 — 정적이므로 등록 안 함
4) main(또는 작업 브랜치) 푸시 → 자동 프리뷰 배포
5) 최종 → Production Promote + 도메인 연결
```
> 🚀 체크포인트마다 푸시만 하면 Vercel이 프리뷰 URL을 줌 → 폰으로도 확인 가능. 별도 배포 명령 불필요.

## 5. 최소 생존 경로 (한도가 빠듯할 때)

한도가 모자라면 **이 순서로 잘라낸다**(뒤에서부터 포기):
1. 반드시: A → B → C → D(case2) → C5 연출 → 배포 = **1케이스 완전체라도 작동**
2. 여유 생기면: F(case1,3) → E2(날씨모션) → G2(대시보드)
3. 남으면: E3·E4(타임머신·단열선도) → G3(AI 피드백)

> 즉 **"작동하는 1케이스"를 항상 먼저 확보**하고, 화려함·확장은 한도 봐가며 쌓는다. 데모가 안 도는 사고만은 절대 안 만든다.
