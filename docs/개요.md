## 0. 한 줄 요약

> **학습자를 기상청 예보관 의자에 앉혀, 진짜 기상청 데이터로 "내일 비 올까?"를 확률로 판단하게 하고,
> 실제 그날 무슨 일이 있었는지 까서 점수를 매기는 웹 체험 도구.**

- 데이터: **100% 실제 기상청 데이터** — 지상·상층 일기도, 위성(3채널), 레이더, ASOS/AWS, 라디오존데, 수치예보(2발표시각), 앙상블, 실황
- 비주얼·해설: **AI 생성** (분위기 일러스트·예보관 캐릭터·챗봇) — 데이터와 명확히 구분 라벨
- 채점: **Brier Score** — 확신하고 틀리면 크게 감점, 애매하면 적당히 (진짜 예보 평가법)
- 분량: **3케이스 난이도 곡선** (쉬움 → 어려움 → 함정) + 성장 대시보드
- 체험 흐름: **6단계** — 실제 기상청 예보관 분석 순서를 그대로 따름

## 1. 무엇을 만드나 — 사용자가 보는 화면 (체험 1회 흐름)

한 케이스는 **6단계**로 흐른다. 케이스당 10~15분. **실제 기상청 예보관 분석 프로세스를 그대로 따른다.**

```
[온보딩 1.5분]  데이터 6종 카드 슬라이드 + 확인 퀴즈
      ↓
① 종관 분석   "2022-08-08 새벽, 너는 서울 예보관"
  (실 예보관   · 지상 일기도 — 전선·저기압·고기압이 어디?
   첫 번째     · 850hPa 바람·온도장 — 저층 수분 유입 방향
   하는 일)    · 500hPa 고도장 — 기압골·제트류 위치
              시간 슬라이더로 -24h → now 흐름 파악 (종관 타임머신)
      ↓
② 현상 분석   지금 실제로 무슨 일이 일어나고 있나?
  (관측망      · 레이더 HSR 타임라인 + 에코 이동벡터 오버레이
   확인)       · 위성 3채널 블렌드 (IR / VIS / 수증기WV 크로스페이드)
              · ASOS 멀티라인 차트 (기온·강수량·기압·습도 타임 동기화)
              · AWS 지점 핀 + 낙뢰 LGT 점멸 맵
      ↓
③ 대기 안정도  비구름이 발달할 조건인가?
  (성층 분석)  · 라디오존데 SKEW-T 인터랙티브 (연직 기온·이슬점·바람 프로파일)
              · CAPE / CIN / K-Index / LI 지수 자동 계산 표시
              · "이 성층이면 뇌우 가능한가?" 안정도 등급 색 표시
      ↓
④ 모델 검토   수치예보가 뭐라고 하나? 얼마나 믿을 수 있나?
  (가이던스    · KIM/UM 두 발표시각 비교 (전날 23시 vs 당일 08시 — 강수확률 다름!)
   검토)       · 앙상블 스프레드 바 (모델들이 얼마나 흩어져 있나?)
              · 불일치 항목 자동 하이라이트
      ↓
⑤ 최종 판단   내가 결정한다
              · 비 올 확률 (0~100%) 슬라이더 → 확신 미터 연동
              · 강수 형태 (비/눈/비눈/없음), 최고기온 (°C)
              · 근거 체크 (①~④ 중 뭘 보고 결정했나) — 필수
      ↓
⑥ 결과 공개   "그날, 실제로는…" → 381.5mm 폭우 공개
   + 채점     → Brier 점수 카운트업 연출 + 반사실 비교
   + 리플렉션 → "무엇을 놓쳤나" 입력 → Claude 맞춤 피드백
      ↓
[대시보드]    3케이스 점수 추세선 → 성장 + "어느 단계에서 잘 봤나" 약점 분석
```

**옆에 항상 떠 있는 것**: 우하단 AI 챗봇(WeatherAssistant). "이 SKEW-T에서 CAPE가 뭔가요?"
물어보면 지금 보는 단계·데이터에 맞춰 2~4문장으로 설명. 정답은 안 알려줌.

### 3티어 케이스 풀 — 세션마다 랜덤 추첨

케이스가 고정이면 두 번째 플레이부터 답을 외울 수 있다. 그래서 **티어별 케이스 풀**을 만들어 세션 시작 시 각 티어에서 1개씩 랜덤으로 뽑는다.

| 티어 | 상황 특징 | 학습 포인트 | 첫 케이스 |
|---|---|---|---|
| easy (쉬움) | 안정적 고기압, 데이터 일치 | 데이터 읽기 기초·성공 경험 | 고기압 맑음 사례 |
| hard (어려움) | 집중호우, 모델이 엇갈림 | 모델 불일치·확률적 사고 | 2022-08-08 수도권 호우 |
| trap (함정) | 모델은 맑음인데 레이더엔 에코 | "데이터를 의심하라"·관측 우선 | 에코 팝업 함정 |

- 순서는 항상 **easy → hard → trap** (난이도 곡선 유지)
- 랜덤은 "어느 사례냐"지 "어떤 순서냐"가 아님
- 다 봤으면 풀 리셋 → 무한 재플레이 가능
- 나중에 케이스 추가 = `case-pool.json`에 id 한 줄만 추가

> 1로 자신감 → 2에서 갈등 → 3에서 반전. 매번 다른 사례로, 같은 구조로.

## 2. 왜 이렇게 만드나 — 핵심 설계 결정 3가지

| 결정 | 내용 | 이유 |
|---|---|---|
| **① 실데이터 정면 사용** | 위성·레이더·관측·예보를 진짜 기상청 API로 수집, 원본 JSON까지 보존 | 심사장에서 "이거 진짜 데이터예요"를 증빙. 주제 적합성 |
| **② AI 비주얼·챗봇 대량 투입** | 케이스당 이미지 15~20장 + 문맥 챗봇 | 딱딱한 데이터를 몰입형으로. 독창성·참여형 |
| **③ 3케이스 완전체** | 얕은 4개 대신, 난이도 곡선 3개를 완성도 100%로 | "원트릭 장난감"이 아닌 "커리큘럼"으로 보이게 |

### 두 가지 똑똑한 트릭

1. **"모델 불일치"를 공짜로 만든다**
   KIM/UM 같은 수치모델 원시 격자는 일반 참가자가 못 받는다. 대신 **같은 날짜를 두 발표시각으로
   호출**한다. 8/7 23시 발표와 8/8 08시 발표가 같은 시점을 다르게 예측한 차이 = "가이던스가 흔들린
   순간". 이건 100% 진짜 기상청 데이터다.

2. **모든 데이터를 미리 받아 정적 파일로 박제**
   런타임에 API를 안 부른다. 현장(천안) 네트워크가 끊겨도 무중단. CDN 캐시로 빠름. API 한도 걱정 0.

### 챗봇은 언제? — G3.5 세션

`api/reflect.js`(리플렉션) 만든 직후, **G3.5** 세션 하나로 챗봇을 붙인다.
패턴이 같아서 짧게 끝남. (`WeatherAssistant.tsx` + `api/chat.js`)

### 최소 생존 경로 (한도가 빠듯하면)

> **A → B → C → D(case2) → 결과 연출 → 배포 = "1케이스라도 완전히 작동"**

## 5. 점수는 어떻게 매기나 — Brier Score

```
강수확률 (60%):  Brier = (내확률/100 − 실제0or1)²  →  점수 = 100 − round(Brier×100)
강수형태 (20%):  맞으면 100, 틀리면 0
최고기온 (20%):  1°C 빗나갈 때마다 −10점
```

| 내 예측 | 실제 | 점수 | 교훈 |
|---|---|---|---|
| "90% 비" | 비 옴 | ~99점 | 확신하고 맞으면 최고점 |
| "90% 비" | 맑음 | ~20점 | **확신하고 틀리면 폭망** |
| "50% 비" | 비 옴 | ~75점 | 애매하게 찍으면 안전빵 |

→ "모르면 50% 찍어"라는 비겁한 전략보다, **데이터로 확신을 높이는 게 이득**임을 점수로 체감시킨다.
대시보드에서 "매번 50%만 찍었다면" 가상선과 비교해 이걸 시각화.

## 6. AI는 어디에 쓰나 (데이터와 명확히 구분)

| 용도 | 무엇 | 원칙 |
|---|---|---|
| 분위기 비주얼 | 브리핑 일러스트·예보관 캐릭터·단계 배경·결과 연출 (케이스당 15~20장) | "분위기·연출"에만 |
| 리플렉션 피드백 | `api/reflect.js` — "무엇을 놓쳤나" 입력에 Claude가 맞춤 피드백 | 실데이터와 대조 |
| **문맥 챗봇** | `api/chat.js` + `WeatherAssistant.tsx` — 지금 보는 데이터를 설명 | **정답(예측 수치)은 안 알려줌** |

> 철칙: **판단 근거가 되는 수치는 100% 실측. AI는 분위기·해설·도우미.**
> 심사장에서 "데이터는 진짜, 비주얼은 AI"를 명확히 구분 설명 → 신뢰성 + 독창성 동시 확보.
