# 「예보관 의사결정 체험」 UX/UI 명세서 (Design Spec v1.0)

> 기상·기후 AI 해커톤 2026 출품작 · **디자이너/프론트엔드 실행용 문서**
> 디자인 언어 기준: 기존 가이드(`index.html`) — Pretendard · 무채색 · 16px 라운드 카드
> 함께 보기: `forecaster-build-spec.md`(구현), `forecaster-app-plan.md`(개요)

## 0. 디자인 명제 (Design Thesis)

이 제품의 디자인은 한 문장 원칙 위에 선다.

> **"UI는 침묵하고, 데이터와 사람의 판단이 말하게 한다."**

- **단정한 무채색 크롬(chrome)** 이 화면의 골격을 잡는다. 버튼·카드·텍스트는 흑백.
- **색은 오직 의미 있는 곳에만** — 레이더의 강수 강도, 점수, 위험도. UI 장식으로 색을 쓰지 않는다.
- **AI 이미지는 "무대 배경"** — 데이터 패널 뒤·전환 화면·결과 연출에 깔되, 판단해야 하는 데이터 위는 침범하지 않는다.

> 이렇게 하면 "화려함"(AI 비주얼)과 "단정함"(UI)이 충돌하지 않는다. **화려한 건 콘텐츠, 단정한 건 조작계.** 둘의 역할을 분리하는 것이 이 디자인의 핵심 전략이다.

### 0.1 4대 UX 원칙
| 원칙 | 의미 | 적용 |
|---|---|---|
| **즉각성** | 조작 → 0.2초 내 화면 반응 | 토글·슬라이더는 입력 즉시 반영, 로딩 스피너 금지(정적) |
| **한 번에 하나** | 한 화면은 한 가지 결정만 요구 | 5단계를 한 페이지에 다 펼치지 않고 단계 전환 |
| **되돌릴 수 있음** | 제출 전까지 모든 입력 수정 가능 | 단계 뒤로가기 허용, 제출만 비가역 |
| **정직한 피드백** | 잘했을 때만이 아니라 틀렸을 때도 명확히 | 낮은 점수도 숨기지 않고 이유를 보여줌 |

## 1. 디자인 토큰 (Design Tokens)

기존 `index.html`의 `:root`를 계승하고 **데이터/점수 의미색만 추가**한다.

### 1.1 색 (Color)
```css
:root {
  /* ── 무채색 크롬 (기존 계승) ── */
  --black:    #111111;
  --gray-900: #1c1c1c;
  --gray-800: #333333;
  --gray-600: #666666;
  --gray-400: #999999;
  --gray-300: #bbbbbb;
  --gray-200: #d9d9d9;
  --gray-100: #eeeeee;
  --gray-50:  #f7f7f7;
  --white:    #ffffff;

  /* ── 의미색: 데이터 (절제해서만) ── */
  --rain:     #2563eb;  /* 강수·비 — 차분한 블루 */
  --rain-soft:#dbeafe;
  --warn:     #e11d48;  /* 위험·호우·강한 에코 — 레드 */
  --warn-soft:#ffe4e6;
  --sun:      #f59e0b;  /* 맑음·고기압 — 앰버 (남용 금지) */

  /* ── 의미색: 점수 ── */
  --score-good: #16a34a;  /* 80점+ */
  --score-mid:  #f59e0b;  /* 50~79 */
  --score-bad:  #e11d48;  /* ~49 */

  /* ── 레이더 강수 강도 스케일 (범례 고정) ── */
  --echo-1:#a5d8ff; --echo-2:#4dabf7; --echo-3:#fab005;
  --echo-4:#f76707; --echo-5:#e03131;
}
```
> **색 사용 규칙**: 한 화면에 의미색은 **최대 2계열**. 점수 화면에서는 score 색, 데이터 화면에서는 rain/warn. 섞지 않는다.

### 1.2 타이포 (Pretendard)
| 토큰 | size / weight / line | 용도 |
|---|---|---|
| `display` | 40px / 800 / 1.1 | 결과 공개 숫자, 히어로 |
| `h1` | 28px / 800 / 1.2 | 단계 제목 |
| `h2` | 20px / 700 / 1.3 | 패널 제목 |
| `body` | 15px / 400 / 1.7 | 본문 |
| `label` | 13px / 600 / 1.4 | 슬라이더·버튼 라벨 |
| `caption` | 12px / 500 / 1.5 | 출처 라벨, 보조설명 |
| `mono` | 13px / SF Mono | 수치값(확률·기온) |

### 1.3 간격·모양·그림자
```
spacing scale: 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 (px)
radius:  버튼 10px · 카드 16px · 패널 20px · 알약(pill) 100px
shadow:  rest  0 1px 2px rgba(0,0,0,0.04)
         hover 0 4px 24px rgba(0,0,0,0.06)
         modal 0 12px 48px rgba(0,0,0,0.12)
border:  1px solid var(--gray-100)  (강조 시 2px var(--black))
```

### 1.4 모션
```
duration:  micro 120ms · base 200ms · enter 320ms · reveal 600ms
easing:    standard cubic-bezier(0.2,0,0,1)
                 emphasized cubic-bezier(0.2,0,0,1.2)  (결과 공개 점수 카운트업)
원칙: 위치/투명도만 애니메이트. 레이아웃 흔드는 전환 금지.
접근성: prefers-reduced-motion 존중 → 모든 애니메이션 0ms로 폴백.
```

## 4. 감정 곡선 (Emotional Arc) — 이 제품 UX의 심장

체험은 **정보 → 갈등 → 결단 → 반전 → 성찰**의 드라마 구조를 따른다. 각 단계의 목표 감정을 디자인이 의도적으로 만든다.

| 단계 | 목표 감정 | 디자인 장치 |
|---|---|---|
| 온보딩 | 호기심 "내가 예보관?" | 예보관 의자에 앉는 1인칭 카피 + 캐릭터 |
| S1 종관 | 몰입 "뭔가 다가온다" | 시간 슬라이더로 에코가 발달하는 걸 직접 재생 |
| S2~3 모델 | **갈등** "어느 쪽을 믿지?" | 두 발표가 엇갈리는 지점을 빨강으로 하이라이트 |
| S4 판단 | **긴장** "내가 결정한다" | 슬라이더를 끝까지 밀 때의 무게감, 제출 전 확인 |
| S5 공개 | **반전** "헉/됐다!" | 0.6초 숫자 카운트업 + 실황 일러스트 페이드인 |
| 성찰 | 깨달음 "다음엔 이걸 본다" | 내 점수 vs 안전 50% 비교선 |

> 결과 공개(S5)는 **즉시 점수를 까지 않는다.** "그날 실제로는…" 1.2초 텀을 두고 실황을 먼저 보여준 뒤 점수가 올라간다. 이 짧은 정적이 반전의 임팩트를 만든다.

## 5. 화면별 와이어프레임 & 상태 (Screen Specs)

### 5.1 랜딩 / 케이스 라이브러리 (`/`)
```
        예보관 의사결정 체험
   불완전한 데이터로 최선의 확률을 판단하는 일

┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ ★★★  HARD   │ │ 🔒 잠김      │ │ 🔒 잠김      │ │ 🔒 잠김      │
│ 수도권      │ │ 오후 소나기 │ │ 태풍 접근   │ │ 함정 케이스 │
│ 집중호우    │ │ (준비중)    │ │ (준비중)    │ │ (준비중)    │
│ 2022.08.08  │ │             │ │             │ │             │
│ [체험 시작]│ │             │ │             │ │             │
└─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
```
- 활성 카드 1개만 클릭 가능, 나머지는 회색 + 자물쇠(확장성 암시).
- 완료 후 재방문 시 활성 카드에 **점수 배지** 노출.
- 카드 hover: shadow-hover + 1px→미세 상승(translateY -2px).

### 5.2 온보딩 (`/onboarding`)
- 카드 4장 가로 스와이프/도트 인디케이터. 각 카드: AI 일러스트 + 한 줄.
  - 위성=구름의 두께·온도 / 레이더=지금 내리는 비 위치 / 모델=앞으로의 예측(불확실) / 지상관측=실제 측정값
- 마지막에 3문항 퀴즈(객관식). **틀려도 통과 가능**(학습용, 게이트 아님) — 오답 시 정답 카드로 부드럽게 되짚어줌.
- 스킵 링크 제공("이미 알아요 →").

### 5.3 S1 종관분석 — LayerViewer
```
┌─ 레이어 ──────────────────────────────────┐
│ [위성] [레이더] [지상관측]      🛈출처 라벨 │  ← 탭(pill), 활성=검정
├────────────────────────────────────────────┤
│                                            │
│        (선택 레이어 이미지/그래프)          │
│        레이더 강수강도 범례 ▮▮▮▮▮          │
│                                            │
├────────────────────────────────────────────┤
│ -24h ●───────────────○ now   [▶ 재생]       │  ← TimeSlider
└────────────────────────────────────────────┘
   안내: "시간을 끌어 에코가 어떻게 변하는지 보세요"
```
- 탭 전환 0.12s 크로스페이드. 시간 슬라이더 드래그 시 프레임 즉시 교체(프리로드 6장).
- 모든 데이터 패널 우상단에 **SourceTag**("기상청 API허브 · 천리안 2A") 상시 노출 → 신뢰성.
- 레이더 범례는 항상 동반(색의 의미를 모르면 데이터가 무의미).

### 5.4 S2~S3 모델 비교 — ModelCompare
```
┌── 8/7 23시 발표 ──┐   ┌── 8/8 08시 발표 ──┐
│ 강수확률  60% ▮▮▮ │   │ 강수확률  90% ▮▮▮▮│ ← 차이나는 값 빨강
│ 강수량   20mm     │   │ 강수량   80mm     │
│ 최고기온  28°C    │   │ 최고기온  26°C    │
└───────────────────┘   └───────────────────┘
        "두 발표가 엇갈립니다. 어느 쪽을 믿나요?"
        ( ) 8/7 발표   (●) 8/8 발표
        이유: [______________________________]
```
- 두 패널 동일 높이, 다른 값만 자동 하이라이트(`--warn`).
- 모바일에서는 상하 적층 + 차이 배지("↑30%").

### 5.5 S4 판단확정 — ForecastInput + Evidence
```
   당신의 예보를 확정하세요

   강수확률   없음 ──────────●── 확실   [ 90% ]
              └ ConfidenceMeter: "꽤 자신있음"
   강수형태   (비) 눈  비/눈  소나기  없음
   최고기온   20 ──────●── 30        [ 26°C ]

   판단 근거 (본 데이터)
   ☑ 레이더   ☑ 지상관측   ☐ 위성   ☑ 모델

              [ 예보 제출하기 ]   ← 근거 1개+ 선택해야 활성
```
- 확률 슬라이더 아래 **ConfidenceMeter**: 50% 근처="애매함", 양 끝="확신" 라벨로 Brier 직관 형성.
- 제출 버튼은 근거 미체크 시 비활성(회색) + 이유 툴팁.
- 제출 직전 **확인 모달**: "제출하면 수정할 수 없어요. 확정?" (비가역 행동 보호).

### 5.6 S5 검증·성찰 — ResultReveal
```
        그날, 실제로는…
   ┌────────────────────────────────┐
   │ (서울 침수 거리 AI 일러스트)     │  ← 1.2s 페이드인
   │                                │
   │   381.5mm 폭우 · 비 · 최고 26°C  │  ← 그 다음 등장
   └────────────────────────────────┘

        당신의 점수      ▲ 0→92 카운트업
   ┌──────────┬──────────┬──────────┐
   │ 강수 99  │ 형태 100 │ 기온 90  │  ← ScoreBreakdown
   └──────────┴──────────┴──────────┘
   "90%로 자신있게 맞혔어요. 안전하게 50%였다면 75점이었죠."

   무엇을 놓쳤나요? [__________________]
                   [ AI 피드백 받기 ]  → Claude
```
- 점수 색은 구간색(good/mid/bad). 카운트업 emphasized easing.
- Brier 교훈을 **반사실(counterfactual)** 한 줄로 항상 보여줌("50%였다면 N점").
- AI 피드백 로딩 동안 스켈레톤(점 3개), 실패 시 "잠시 후 다시" 폴백 카피.

### 5.7 성장 대시보드
- GrowthChart: 케이스가 1개뿐이므로 v1은 **단일 점 + 안내**("케이스가 늘면 추세가 그려집니다"). 가짜 데이터로 채우지 않는다.
- BadgeShelf: 점수→등급(수습/주임/책임/수석/대장 예보관) 일러스트 배지 획득.
- 공유 버튼: 점수 카드 이미지 다운로드(부스 인증샷용).

## 6. 상태 디자인 (모든 컴포넌트 필수)

| 상태 | 규칙 |
|---|---|
| **기본(rest)** | 위 명세 |
| **호버/포커스** | 포커스 링 `2px var(--black) offset 2px`. 키보드 탐색 100% 지원 |
| **로딩** | 정적이라 데이터 로딩 없음. AI 피드백만 스켈레톤(레이아웃 점프 0) |

| **에러** | AI/네트워크 실패 시 기능 차단 말고 **체험은 계속**(피드백만 생략) |
| **빈 상태** | 대시보드 데이터 없음 → 가짜로 채우지 말고 안내 카피 |
| **완료** | 제출/획득 시 체크 마이크로 애니메이션(120ms) |

## 7. 접근성 (WCAG AA 목표)
- 명도 대비: 본문 ≥ 4.5:1, 큰 글자 ≥ 3:1. 무채색이라 유리.
- **색에만 의존 금지**: 모델 차이는 색+화살표 배지, 점수는 색+숫자+라벨 병기.
- 모든 조작 키보드 가능(Tab/방향키로 슬라이더, Enter로 제출).
- 슬라이더 `aria-valuenow/valuetext`("90퍼센트, 꽤 자신있음").
- 이미지 `alt`: 데이터 영상은 설명 alt, AI 장식은 `alt=""`.
- `prefers-reduced-motion` 시 카운트업·페이드 즉시 표시.
- 터치 타깃 ≥ 44×44px(키오스크·모바일).

---

## 8. 보이스 & 마이크로카피
- **톤**: 차분한 선배 예보관. 어린이 말투(X), 전문 용어 던지기(X). "쉽게, 그러나 가볍지 않게."
- 1인칭 몰입: "당신의 예보를", "무엇을 놓쳤나요".
- 숫자엔 의미를 붙임: "381.5mm — 한 달 치 비가 하루에".
- 실패 카피는 격려가 아니라 **학습 포인트**로: "틀린 게 아니라, 레이더를 덜 봤어요."
- 금지: 느낌표 남발, "축하합니다🎉" 류 과장. 단정함 유지.

## 9. AI 비주얼 ↔ UI 충돌 방지 규칙
1. AI 이미지는 **배경/전환/결과**에만. 데이터·슬라이더·숫자 위에 겹치지 않는다.
2. 배경으로 쓸 땐 opacity ≤ 0.12 또는 상단 화이트 그라데이션으로 가독성 확보.
3. AI 이미지엔 항상 미세한 "AI 생성" 캡션(좌하단 10px) — 실데이터와 구분(신뢰성).
4. 이미지 스타일 통일: 동일 일러스트 톤(채도·붓터치) 한 세트로 생성, 사진/3D 혼용 금지.
5. 로딩 체감 0: WebP + 단계별 프리로드(다음 단계 배경 미리).

## 11. 안티-제너릭 — "AI가 만든 웹" 냄새 제거 전략

바이브 코딩으로 만들면 십중팔구 **"AI 웹사이트 룩"**이 나온다. 심사위원도 한눈에 알아챈다.
이 섹션은 그 냄새를 의도적으로 제거하는 규칙이다. 원인은 하나 — **의견 없는 디자인**.
해법도 하나 — **무채색 에디토리얼이라는 우리 의견을 끝까지 밀어붙인다.**

### 11.1 즉시 식별되는 AI 웹 냄새 (금지 목록)
| 냄새 | 왜 AI 티가 나나 | 우리의 금지 규칙 |
|---|---|---|
| 보라/인디고 그라데이션 | ChatGPT·v0 기본 출력의 상징 | 그라데이션 자체 금지. 색은 §1.1 의미색만 |
| 똑같은 3등분 feature 카드 | 템플릿 기본형 | 카드 열 수를 의도적으로 비대칭(1·4·잠금3 등) |
| 가운데 정렬 히어로 + 큰 버튼 2개 | SaaS 랜딩 클리셰 | 좌측 정렬 에디토리얼 헤드라인, CTA 1개 |
| 이모지 불릿(🚀✨🎯) | 생성형의 습관 | 이모지 0개. 번호/대시/아이콘(SVG)만 |
| 글래스모피즘 + blur 블롭 | 2023 AI 템플릿 잔재 | 반투명 카드·떠다니는 블롭 금지 |
| 라운드 천국(모든 게 둥금) | 차별성 없음 | radius 위계 고정(버튼10·카드16·패널20), 데이터 패널은 각진 4px |
| Inter/시스템 폰트 그대로 | 무개성 | Pretendard 고정 + 수치는 mono 병행(개성) |
| 완벽 대칭·균일 여백 | 기계적 | 의도된 비대칭(§11.3), 여백에 리듬 |
| 추상 3D 일러스트 히어로 | 스톡 AI 이미지 | 히어로는 실데이터(레이더 영상) 또는 절제된 일러스트 |
| 무미건조한 카피 | 보이스 없음 | 1인칭 예보관 보이스(§8), 도메인 어휘 사용 |

### 11.2 차별화의 정체 — "관제실(管制室) 에디토리얼"
이 사이트만의 시각 모티프를 하나 못박는다. **예보 당직실의 작업 문서** 미감.
- **모노스페이스 데이터 리드아웃**: 확률·기온·강수량은 항상 `mono` 폰트 + 단위. 숫자가 "측정값"처럼 보인다.
- **괘선(罫線) 그리드**: 데이터 패널에 옅은 격자선(graph paper 느낌)으로 "관측 차트" 분위기.
- **타임스탬프 노출**: 모든 데이터에 `2022-08-08 06:00 KST` 같은 실제 관측시각을 작게. 진짜 운영 화면 같은 신뢰감.
- **흑백 + 단 하나의 데이터 색**: 화면당 의미색 1계열. 이 절제가 곧 개성.
> 이 모티프는 AI 템플릿엔 절대 없다. "이건 누가 의도를 갖고 디자인했다"는 신호가 된다.

### 11.3 의도된 비대칭 (Anti-symmetry)
AI는 모든 걸 가운데·균등 배치한다. 우리는 의도적으로 깬다.
- 랜딩 헤드라인 좌측 정렬, 우측은 비움 또는 작은 메타.
- 단계 화면: 데이터 패널(넓게) + 조작계(좁게)의 황금비(≈62:38) 2단.
- 카드 그리드는 균등 3·4열 대신 **1개 대형(활성) + 3개 소형(잠금)**.
- 섹션 간 여백을 일정하게 두지 말고 리듬을 줌(48 / 64 / 48 / 96).

### 11.4 인간의 손맛 (의도적 불완전)
- 수치를 깔끔한 반올림으로만 두지 않음: `381.5mm`(실측 그대로)가 `380mm`보다 진짜다.
- 도메인 정확 어휘: "정체전선", "에코", "HSR 합성영상", "108 서울 지점" — 일반 생성형은 못 쓰는 디테일.
- 마이크로카피에 관점: "틀린 게 아니라 레이더를 덜 봤어요"(§8) — 템플릿엔 없는 보이스.

### 11.5 바이브 코딩 가드레일 (AI에게 시킬 때 반드시 주입)
프롬프트에 매번 붙일 제약. 이게 없으면 AI는 기본형으로 회귀한다.
```
- 디자인 토큰(§1.1)만 사용. 임의 색·그라데이션 생성 금지.
- 보라/인디고 그라데이션, 글래스모피즘, 이모지 불릿 금지.
- 폰트는 Pretendard, 수치는 monospace. 다른 폰트 도입 금지.
- 가운데 정렬 히어로·3등분 카드·떠다니는 블롭 만들지 말 것.
- 레이아웃은 의도된 비대칭(§11.3)을 따를 것.
- 카피는 1인칭 예보관 보이스(§8), 이모지·과장 금지.
```
> **검수 한 줄 테스트**: "이 화면을 다른 기상 사이트로 바꿔치기해도 아무도 모를까?"
> 그렇다면 보이스가 없는 것 — §11.2 모티프를 더 입힌다.

  [Delv: AI images that don't look AI](https://delv.tools/blog/ai-images-that-dont-look-ai-generated),
