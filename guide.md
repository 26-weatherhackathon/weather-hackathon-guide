# 기상·기후 AI 해커톤 2026 참가자 가이드

설치도 터미널도 없이, 웹 Claude Code에게 한국어로 시키면 코드 작성부터 GitHub 저장, Vercel 배포까지 이어집니다. 이 문서는 그 전 과정을 단계별로 안내합니다.

무엇을 만드나요? 기상·기후 데이터를 학습자가 직접 조작하며 배우는 웹 도구입니다. 예보관 체험, 기압·바람 인터랙티브 맵처럼요. 서버도 앱도 필요 없이 웹페이지 하나로 충분합니다.

진행 순서
1. 아이디어: 기상 데이터 + 체험 설계
2. 제작: 웹 Claude Code로 코드 작성 + GitHub 자동 저장
3. 배포: Vercel로 웹 링크 배포 후 제출

## 1. 시작 전에 (용어 & 준비물)

아래 용어 5개만 알면 끝까지 따라올 수 있습니다. 지금 다 외울 필요는 없고, 막히면 돌아오면 됩니다.

### 1.1 용어 사전

1.1.1 웹 Claude Code (claude.ai/code)
설치 없이 웹브라우저에서 대화만으로 코딩하는 AI 도구입니다. 한국어로 "이런 웹페이지 만들어줘"라고 요청하면 코드를 대신 작성하고, 연결된 GitHub 저장소에 자동으로 저장까지 해줍니다.
비유: 코드를 대신 써주고 정리·저장까지 해주는 만능 비서. 말로 설명만 하면 됩니다.

1.1.2 index.html (정적 웹페이지)
브라우저에서 바로 열리는 웹페이지 파일 한 개입니다. 글, 그림, 버튼, 슬라이더 같은 인터랙션이 이 안에 다 들어갑니다.
비유: 한 장짜리 완성된 전단지. 이 파일만 있으면 웹사이트가 됩니다.

1.1.3 GitHub (Repository, 저장소)
코드 파일을 인터넷에 보관하는 창고입니다. 저장소(repository)는 그 안의 프로젝트 폴더 하나입니다. 웹 Claude Code가 만든 코드는 여기에 자동으로 저장됩니다.
비유: 코드를 올려두는 구글 드라이브. 저장소 = 폴더 하나.

1.1.4 Vercel (배포, deploy)
GitHub에 올린 코드를 실제 인터넷 주소(URL)로 만들어 누구나 접속하게 해주는 서비스입니다. 이 과정을 배포라고 합니다.
비유: 내 전단지를 거리에 붙여서 모두가 볼 수 있게 하는 것.

1.1.5 PR (Pull Request)
내 브랜치의 변경 내용을 팀의 메인 코드에 합치자고 요청하는 것입니다. 팀원이 검토(리뷰)하고 OK하면 합쳐집니다. 자세한 내용은 3장을 참고하세요.
비유: "이 부분 이렇게 바꿨는데, 본책에 넣어도 될까요?" 하고 올리는 검토 요청서.

### 1.2 준비물 (모두 무료)

계정 3개만 만들어 두세요. 설치할 프로그램은 없고, 웹브라우저만 있으면 됩니다.

1.2.1 Claude 계정
Claude Code 사용. claude.ai/code — 본선팀 제공 플랜으로 로그인.

1.2.2 GitHub 계정
코드 저장소. github.com에서 가입 — Claude Code와 연결합니다.

1.2.3 Vercel 계정
배포 서비스. vercel.com — GitHub 계정으로 바로 로그인 가능.

설치할 게 없습니다. 터미널, Node.js 같은 프로그램 설치는 필요 없어요. 세 사이트 모두 웹브라우저에서 로그인만 하면 끝입니다.

## 2. 만들기 (Claude Code로 제작·배포)

만들고 싶은 걸 한국어로 설명하면 Claude가 코드를 짜고 GitHub에 저장합니다. 아래 5단계를 따라오세요.

### 2.1 Step 1 — 웹 Claude Code 시작하기

2.1.1 GitHub 저장소 먼저 만들기
1. github.com 로그인 후 우측 상단 + → New repository
2. 이름 입력 (예: weather-project), Private 선택
3. Create repository 클릭

저장소 이름은 나중에 Vercel 주소가 됩니다. 팀 프로젝트 이름으로 지어두세요.

2.1.2 claude.ai/code 접속 후 GitHub 연결
1. claude.ai/code 접속 → 제공받은 계정으로 로그인
2. 입력창 아래쪽의 "+ 레포 선택..." 버튼 클릭
3. 목록에서 방금 만든 저장소(예: weather-project) 선택
4. 선택이 끝나면 버튼 자리에 저장소 이름과 브랜치 이름(예: master)이 표시됩니다

이 연결을 한 번만 해두면 이후 Claude Code가 코드를 짤 때마다 GitHub에 자동으로 저장합니다.

### 2.2 Step 2 — Claude Code로 만들기

2.2.1 한국어로 설명하면 코드가 나온다
요청할 때 (1) 무엇을 보여줄지 (2) 어떻게 조작할지 (3) 파일 이름(index.html), 이 3가지만 넣으면 됩니다.

예시 대화
- 사용자: 기온과 풍속 슬라이더로 체감온도를 보여주는 웹페이지를 index.html 하나로 만들어줘. 기상청 체감온도 공식을 써줘.
- Claude: 슬라이더를 움직이면 체감온도가 실시간으로 바뀌는 index.html을 만들었습니다. GitHub에 저장했어요.
- 사용자: 배경색을 하늘색으로 바꾸고, 모바일에서도 잘 보이게 해줘.
- Claude: 배경색과 반응형 레이아웃을 적용했습니다. 변경 사항을 GitHub에 저장했어요.

2.2.2 바로 쓰는 프롬프트 모음
대괄호 부분만 우리 팀 주제로 바꾸세요.
- 기본형: "[기온]과 [풍속] 슬라이더로 [체감온도]를 보여주는 웹페이지를 index.html 하나로 만들어줘."
- 응용형: "[강수량] 슬라이더를 조절하면 [홍수 위험도]가 바뀌는 인터랙티브 웹페이지를 초등학생도 이해하게 만들어줘. index.html 하나로."
- 디자인: "방금 페이지를 더 보기 좋게 꾸며줘. 파란 하늘색 계열, 모바일에서도 잘 보이게."
- 오류 해결: "방금 페이지에서 [오류 메시지]가 나와. 원인을 알려주고 고쳐줘."

팁: 한 번에 완벽하지 않아도 됩니다. "여기를 이렇게 바꿔줘"라고 계속 대화하며 다듬으세요. 수정할 때마다 GitHub에 자동 저장됩니다.

### 2.3 Step 3 — CLAUDE.md 설정하기

2.3.1 프로젝트 규칙서, CLAUDE.md
CLAUDE.md는 Claude Code가 프로젝트를 열 때 가장 먼저 읽는 파일입니다. 여기 써둔 규칙은 매 대화마다 자동 적용됩니다. "파일 이름은 index.html로 고정", "한국어로 응답" 같은 규칙을 넣어두면 매번 설명할 필요가 없어요.

설정 방법
1. 아래 CLAUDE.md 내용을 전체 복사합니다.
2. GitHub 저장소에서 Add file → Create new file 클릭
3. 파일 이름을 CLAUDE.md로 입력하고 붙여넣기 → Commit changes

파일 이름은 반드시 CLAUDE.md(대문자), 위치는 저장소 최상단 폴더입니다.

CLAUDE.md (복사용)
```markdown
# CLAUDE.md

## Git 규칙 (필수)
- main 브랜치에 직접 푸시하지 않습니다. 작은 수정도 예외 없음.
- 모든 변경은 새 브랜치 → PR(Pull Request)로 머지합니다.
  - 팀원이 있으면: 팀원 1명 이상이 리뷰·승인한 뒤 머지
  - 혼자(1인 팀)면: PR을 올려 변경 내용(diff)을 스스로 확인한 뒤 머지
- main에는 항상 정상 동작하는 코드만 둡니다.
```

### 2.4 Step 4 — Vercel로 배포하기

2.4.1 GitHub로 로그인 후 프로젝트 가져오기
1. vercel.com 접속 → Continue with GitHub
2. Add New → Project 클릭
3. 방금 만든 저장소 선택 → Import

2.4.2 Application Preset 설정
Import 직후 설정 화면에서 Framework Preset을 프로젝트에 맞게 선택합니다.

| 내 프로젝트 유형 | 선택할 Preset |
|---|---|
| HTML/CSS/JS 파일만 (index.html 중심) | Other |
| Astro로 만든 경우 | Astro |
| Next.js로 만든 경우 | Next.js |
| Vite로 만든 경우 | Vite |

모르겠으면 Other를 선택하세요. index.html을 직접 만들었다면 Other입니다. Preset이 틀리면 빌드가 실패합니다.

2.4.3 배포 후 링크 받기
1. 설정 확인 후 Deploy 클릭
2. 1~2분 뒤 weather-project.vercel.app 같은 주소 생성
3. 이 주소가 제출용 링크입니다

편한 점: GitHub에서 index.html을 수정하면 Vercel이 자동으로 다시 배포합니다. 따로 재배포할 필요 없어요.
배포 후 확인: 휴대폰이나 다른 컴퓨터에서도 링크가 열리는지, 슬라이더 등 인터랙티브 기능이 잘 작동하는지 꼭 확인하세요.

### 2.5 Step 5 — 제출하기

2.5.1 제출 항목 3가지
Vercel 링크 하나만 있으면 안 됩니다. 아래 3가지를 모두 준비하세요.
- Vercel 배포 URL: 누구나 접속할 수 있는 웹 주소 (예: weather-project.vercel.app)
- GitHub 저장소 URL: 코드가 저장된 주소. 커밋 히스토리가 남아 있어야 합니다.
- Claude 대화 내역: AI 활용 내역 확인용. 운영팀 안내에 따라 세션 링크 또는 스크린샷 첨부.

GitHub 커밋 히스토리는 표절·저작권 검증에 사용됩니다. 개발 과정 전체가 남도록 중간중간 저장하세요.

2.5.2 제출 전 체크리스트
- Vercel URL이 다른 기기에서도 열리는가
- GitHub 저장소가 Private으로 설정되어 있는가
- 커밋 히스토리가 개발 과정 전체를 포함하는가
- 외부 리소스(이미지·폰트·오픈소스)의 출처·라이선스를 README에 명시했는가

## 3. 팀 협업 (GitHub PR)

최종 소스 코드는 Google Drive에 업로드해 제출합니다. 아래 GitHub PR 협업 방법은 팀(2인 이상)이 서로 방해 없이 함께 개발할 때를 위한 안내입니다. 1인 참가자라면 이 장은 참고만 하면 됩니다. 혼자 작업할 땐 브랜치·PR 없이 그냥 개발하고, 완성된 소스 코드를 Google Drive에 올리면 됩니다.

PR이란? Pull Request — 내가 만든 브랜치의 변경 내용을 팀의 메인 코드에 합치자고 요청하는 것입니다. 팀원이 검토하고 OK하면 합쳐집니다. 충돌 없이 협업하는 표준 방식입니다.

팀원이 각자 다른 기능을 개발할 땐 브랜치를 나눠 작업하고 PR로 합칩니다. Claude Code에게 말로 시키면 브랜치 생성부터 PR까지 전부 대신 해줍니다.

### 3.1 브랜치 만들기
기능 개발을 시작할 때 main과 분리된 브랜치를 먼저 만듭니다. 각자 다른 브랜치에서 작업하면 서로 방해받지 않습니다.
프롬프트 예시: "새 브랜치를 만들어줘. 브랜치 이름은 feature/wind-chart 로 해줘."
브랜치 이름 규칙: feature/기능명 형태가 일반적입니다. 예: feature/typhoon-map, feature/rainfall-slider, fix/api-error

### 3.2 기능 개발하기
브랜치 위에서 기능을 개발합니다. 평소처럼 Claude Code에게 말로 요청하면 됩니다.
프롬프트 예시: "바람 방향을 0~360도 슬라이더로 조절하면 화살표가 회전하는 인터랙티브 컴포넌트를 만들어줘. index.html에 추가해줘."

### 3.3 커밋하기
작업한 내용을 중간중간 저장(커밋)합니다. 커밋은 "이 시점까지의 스냅샷"입니다. 문제가 생기면 이 시점으로 되돌아올 수 있습니다.
프롬프트 예시: "지금까지 변경한 내용을 커밋해줘. 메시지는 '바람 방향 화살표 컴포넌트 추가'로 해줘."
팁: 큰 작업이 끝날 때마다 커밋하세요. 메시지는 "무엇을 했는지" 한 줄로. 커밋 히스토리는 심사·표절 검증에 활용됩니다.

### 3.4 PR 올리기
기능 개발이 끝나면 검토를 요청하는 PR을 올립니다. Claude Code가 제목과 설명까지 자동 작성해줍니다.
프롬프트 예시: "feature/wind-chart 브랜치를 main에 합치는 PR을 올려줘. 제목은 '바람 방향 시각화 기능 추가'로 해줘."

### 3.5 리뷰 요청하기
PR을 올린 뒤 팀원에게 리뷰를 요청합니다. Claude Code에게 팀원 GitHub 아이디를 알려주면 리뷰어로 지정해줍니다.
프롬프트 예시: "이 PR에 OOO님 리뷰를 요청해줘."
또는 GitHub 웹에서 직접 요청하기
1. GitHub에서 PR 페이지를 엽니다
2. 오른쪽 사이드바 Reviewers 옆 톱니바퀴 클릭
3. 드롭다운에서 리뷰를 요청할 팀원 선택
4. 선택 즉시 그 팀원에게 리뷰 요청 알림이 전송됩니다

### 3.6 리뷰 반영하기
팀원이 수정 요청(리뷰)을 남기면, 그 내용을 Claude Code에게 전달해 반영합니다. 리뷰 내용은 GitHub PR 페이지에서 확인한 뒤, 그대로 복사해서 Claude Code에게 전달하면 됩니다.

### 3.7 머지(Merge)하기
리뷰가 완료되고 승인을 받으면 main 브랜치에 합칩니다. 머지 후 Vercel이 자동으로 최신 버전을 다시 배포합니다. 머지 후 배포 주소를 다시 열어 기능이 정상 동작하는지 꼭 확인하세요.

### 3.8 트러블슈팅

3.8.1 충돌(Conflict)이 생겼을 때
두 팀원이 같은 파일의 같은 부분을 동시에 수정하면 충돌이 생깁니다. 겁먹지 말고 Claude Code에게 그대로 알려주면 됩니다. 어느 쪽을 유지할지, 두 내용을 합칠지 물어보면 처리해줍니다.

3.8.2 Push할 때 SSH 키 에러가 날 때
새 브랜치를 올릴 때 "Permission denied (publickey)" 에러가 날 수 있습니다. 웹 Claude Code는 개인 SSH 키를 쓰지 않고 GitHub App 연결로 인증하므로, 이 에러는 거의 항상 App 설치·권한 문제입니다.
해결: GitHub 프로필 → Settings → Applications → Installed GitHub Apps → Claude의 Configure에서, 작업 중인 저장소가 Repository access에 포함됐는지, 권한이 Contents: Read and write / Pull requests: Read and write로 되어 있는지 확인한 뒤, claude.ai/code에서 저장소를 다시 선택합니다.

팀 작업 권장 흐름: 기능별로 브랜치를 나눈다 → 각자 개발 후 PR → 팀원 1명 이상 리뷰 → 머지 → Vercel 자동 배포. main 브랜치에는 항상 동작하는 코드만 유지하는 게 원칙입니다.

## 4. 규정 (저작권·보안·심사)

결과물은 기상청에서 활용될 수 있습니다. 라이선스 문제가 있으면 수상이 취소될 수 있으니 기준을 꼭 확인하세요. 그리고 만들 때 배점을 떠올리면 방향 잡기가 쉽습니다.

### 4.1 저작권 가이드라인

제출 코드는 팀 간·외부 출처 표절 검증, GitHub 커밋 히스토리 확인, Claude 대화 세션 확인으로 검증됩니다. AI 사용 자체는 문제없지만, 다른 팀이나 외부 출처 코드를 그대로 쓰는 건 허용되지 않습니다.

| 항목 | 허용 | 주의 필요 |
|---|---|---|
| AI 생성 코드 | Claude Code로 생성한 코드 사용 OK | 다른 팀과 동일한 코드 그대로 제출 금지 |
| 오픈소스 라이브러리 | MIT, Apache 2.0 등 상업적 허용 라이선스 | GPL 계열은 기상청 활용 시 제약. README 명시 필요 |
| 이미지·아이콘 | CC0, Unsplash, 직접 제작 | 출처 불명 이미지, 유료 스톡 무단 사용 금지 |
| 폰트 | Google Fonts, Pretendard, Noto Sans 등 무료 폰트 | 상업용 폰트 무단 사용 금지. 라이선스 README 명시 |
| 기상청 데이터 | 공공데이터포털(data.go.kr) 공개 API | 원출처 표기 필수 |

README에 꼭 써야 할 것: 사용한 외부 리소스 목록(라이브러리명·버전·라이선스) / 이미지·폰트 출처 / 기상청 데이터 출처 표기.

### 4.2 API 키 보안 (필수)

API 키가 노출되면 실격입니다. API 키·토큰·비밀번호 같은 비밀값은 어떤 경우에도 코드에 그대로 노출되지 않도록 설계해야 합니다. GitHub에 올라간 코드와 배포된 웹페이지는 누구나 열어볼 수 있어, 노출된 키는 도용·과금 사고로 이어집니다. 키 노출이 확인되면 실격 처리됩니다.

| 하지 말 것 | 이렇게 설계하세요 |
|---|---|
| 키를 HTML·JS 코드에 직접 써넣기 | 키가 필요 없는 공개 데이터·API를 우선 사용 |
| 키를 GitHub에 커밋 | 키가 꼭 필요하면 코드가 아닌 환경변수에 두고, 서버(서버리스 함수)를 거쳐 호출 |
| 키를 Slack·PR·스크린샷에 노출 | 키에 사용량·도메인 제한을 걸어 피해 범위를 최소화 |
| 노출된 키를 그대로 두기 | 실수로 올렸다면 즉시 키를 폐기(revoke)하고 새로 발급 |

막혔을 때: "이 API 키가 코드에 노출되지 않게 안전하게 쓰는 방법 알려줘"라고 Claude Code에게 물어보면 환경변수 설정과 호출 방식까지 안내해 줍니다.

## 5. 자주 묻는 질문 (FAQ)

5.1 코딩을 전혀 못하는데 가능한가요?
가능합니다. 웹 Claude Code에 한국어로 요청하면 코드를 대신 짜줘요. 이 가이드 순서대로만 따라 하면 됩니다.

5.2 백엔드 서버가 꼭 필요한가요?
아닙니다. 브라우저에서 동작하는 웹페이지(index.html) 하나로 충분합니다.

5.3 팀원과 함께 작업하려면?
GitHub 저장소 Settings → Collaborators에서 팀원을 추가하면 됩니다. 각자 Claude Code에서 같은 저장소에 연결해 작업할 수 있어요. 자세한 협업 흐름은 3장 팀 협업을 보세요.

5.4 기술·도구 문의는 어디에 하나요?
Slack의 #질문-공통 채널에 올려주세요. 운영팀이 직접 응대합니다. 정규 응대일은 매주 공지를 확인해주세요.

5.5 Claude Code 계정은 어떻게 받나요?
본선 진출 15팀에게 운영팀이 이메일로 계정을 발급합니다. 계정 접속 문제는 Slack #질문-공통 채널 또는 긴급 시 운영팀에 직접 연락하세요.

5.6 AI가 만든 코드를 그대로 제출해도 되나요?
됩니다. 이 대회는 AI 활용을 권장해요. 단, 다른 팀이나 외부 출처 코드를 그대로 복사하는 건 표절로 처리됩니다. AI로 생성했더라도 우리 팀의 아이디어와 구성이 담겨 있어야 합니다.

아직 안 풀렸다면 Slack #질문-공통에 질문을 남겨주세요.
