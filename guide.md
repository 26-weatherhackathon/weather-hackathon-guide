# 기상·기후 AI 해커톤 2026 참가자 가이드

이 문서는 초보자용 웹 인터페이스를 기준으로 합니다. 웹 Claude Code에 한국어로 요청하면 코드 작성부터 배포까지 브라우저에서 진행할 수 있으며, 터미널·IDE 환경으로 작업해도 무방합니다. 어느 인터페이스로 만들든 제출물 요건만 충족하면 됩니다.

무엇을 만드나요? 기상·기후 데이터를 학습자가 직접 조작하며 배우는 웹 도구를 만듭니다. 예를 들어 예보관 체험이나 기압·바람 인터랙티브 맵과 같은 형태입니다.

**진행 순서**

1. 아이디어: 기상 데이터와 체험 설계
2. 제작: Claude Code로 코드 작성
3. 배포·제출: 웹으로 배포한 뒤 구동 URL·소스·프롬프트 세션·문서를 제출 (자세한 내용은 2장)

---

## 1. 시작 전에 (용어 & 준비물)

가이드에 자주 나오는 핵심 용어 5개를 먼저 정리했습니다. 지금 모두 외울 필요는 없으며, 막히면 다시 돌아와 확인하면 됩니다.

### 1.1 용어 사전

**1.1.1 웹 Claude Code (claude.ai/code)**
설치 없이 웹브라우저에서 대화만으로 코딩하는 AI 도구입니다. 한국어로 "이런 웹페이지를 만들어 달라"고 요청하면 코드를 대신 작성합니다. 처음에 GitHub 저장소를 한 번 연결해 두면(2장 Step 1), 이후 작성한 코드가 그 저장소에 자동으로 저장됩니다.
비유하자면, 요청한 내용을 코드로 작성하고 저장까지 대신 처리해 주는 도구입니다.

**1.1.2 index.html (정적 웹페이지)**
브라우저에서 바로 열리는 웹페이지 파일 한 개입니다. 글, 그림, 버튼, 슬라이더 같은 인터랙션이 모두 이 안에 들어갑니다.
비유하자면, 한 장으로 완성된 전단지입니다. 이 파일만 있으면 웹사이트가 됩니다.

**1.1.3 GitHub (Repository, 저장소)**
코드 파일을 인터넷에 보관하는 창고입니다. 저장소(repository)는 그 안의 프로젝트 폴더 하나에 해당합니다. 2장 Step 1에서 저장소를 만들어 웹 Claude Code에 연결하면, 이후 작성한 코드가 이 저장소에 저장됩니다. 이 저장소는 개발용 작업 공간이며, 최종 소스 제출은 저장소 링크가 아니라 별도 구글 폼에 업로드합니다(2장 Step 5).
비유하자면, 코드를 올려 두는 온라인 저장 공간이며, 저장소는 그 안의 폴더 하나입니다.

**1.1.4 Vercel (배포, deploy)**
GitHub에 올린 코드를 실제 인터넷 주소(URL)로 만들어 누구나 접속할 수 있게 하는 서비스입니다. 이 과정을 배포라고 합니다.
비유하자면, 완성한 전단지를 거리에 붙여 누구나 볼 수 있게 하는 것과 같습니다.

**1.1.5 PR (Pull Request)**
내 브랜치의 변경 내용을 팀의 메인 코드에 합치자고 요청하는 것입니다. 팀원이 검토(리뷰)한 뒤 승인하면 합쳐집니다. 자세한 내용은 3장을 참고하세요.
비유하자면, "이 부분을 이렇게 바꿨는데 반영해도 될지" 확인을 요청하는 검토 요청서입니다.

### 1.2 준비물

계정 3개를 미리 만들어 두세요. 웹 인터페이스에서는 설치할 프로그램이 없으며, 웹브라우저만 있으면 됩니다.

**1.2.1 Claude 계정**
Claude Code 사용. claude.ai/code — 본선 진출 팀에 운영팀이 발급하는 팀 계정으로 로그인합니다.

**1.2.2 GitHub 계정**
코드 저장소. github.com에서 가입한 뒤 Claude Code와 연결합니다.

**1.2.3 Vercel 계정**
배포 서비스. vercel.com — GitHub 계정으로 바로 로그인할 수 있습니다.

Claude Code 웹 인터페이스는 설치가 없습니다. 초보자용 Claude Code 웹 인터페이스에서는 터미널이나 Node.js 설치가 필요하지 않으며, 세 사이트 모두 브라우저 로그인만으로 시작합니다. 터미널·IDE 환경에 익숙하다면 그 환경으로 작업해도 됩니다.

## 2. 만들기 (Claude Code로 제작·배포)

만들고자 하는 내용을 한국어로 설명하면 Claude가 코드를 작성하고 작업 저장소(GitHub)에 저장합니다. 아래 5단계는 초보자용 웹 인터페이스를 기준으로 하며, 터미널·IDE로 작업해도 됩니다.

### 2.1 Step 1 — 웹 Claude Code 시작하기

웹 Claude Code를 GitHub 저장소에 연결하여 동작하는 것을 가정하여 설명합니다. 이 저장소는 개발용 작업 공간이며 반드시 최종 소스는 GitHub이 아닌 구글 폼에 업로드합니다.

**2.1.1 GitHub 저장소 먼저 만들기**

1. github.com 로그인 후 우측 상단 + → New repository
2. 이름 입력(예: weather-project) — 공개/비공개(Public/Private)는 자유롭게 선택
3. Create repository 클릭

저장소 이름은 이후 Vercel 주소가 되므로, 팀 프로젝트 이름으로 지어 두면 좋습니다.

**2.1.2 claude.ai/code 접속 후 GitHub 연결**

1. claude.ai/code 접속 → 제공받은 계정으로 로그인
2. 입력창 아래쪽의 "+ 레포 선택..." 버튼 클릭
3. 목록에서 방금 만든 저장소(예: weather-project) 선택
4. 선택이 끝나면 버튼 자리에 저장소 이름과 브랜치 이름이 표시됩니다. 브랜치 이름은 계정 설정에 따라 main 또는 master로 보일 수 있으며, 둘 다 '기본 브랜치'라는 같은 의미이므로 그대로 두면 됩니다.

이 연결을 한 번 해 두면 이후 Claude Code가 코드를 작성할 때마다 GitHub에 자동으로 저장합니다.

연결·Push 에러가 날 때(개인·팀 공통): 저장(푸시)하거나 새 브랜치를 올릴 때 "Permission denied (publickey)" 에러가 나거나 저장소 목록이 보이지 않을 수 있습니다. 웹 Claude Code는 개인 SSH 키가 아니라 Claude GitHub App(github.com/apps/claude) 연결로 인증하므로, 대부분 App 설치·권한 문제입니다. GitHub 프로필 → Settings → Applications → Installed GitHub Apps → Claude의 Configure에서 작업 저장소가 Repository access에 포함됐는지, 권한이 Contents: Read and write / Pull requests: Read and write인지 확인한 뒤 claude.ai/code에서 저장소를 다시 선택하세요. 혼자 작업할 때도 동일하게 발생할 수 있습니다.

### 2.2 Step 2 — Claude Code로 만들기

**2.2.1 자연어로 요청하면 코드가 작성됩니다**
요청할 때 (1) 무엇을 보여줄지 (2) 어떻게 조작할지 (3) 파일 이름(index.html), 이 세 가지만 포함하면 됩니다.

예시 대화

- 사용자: 기온과 풍속 슬라이더로 체감온도를 보여주는 웹페이지를 index.html 하나로 만들어 줘. 기상청 체감온도 공식을 사용해 줘.
- Claude: 슬라이더를 움직이면 체감온도가 실시간으로 바뀌는 index.html을 만들었습니다. GitHub에 저장했습니다.
- 사용자: 배경색을 하늘색으로 바꾸고, 모바일에서도 잘 보이게 해 줘.
- Claude: 배경색과 반응형 레이아웃을 적용했습니다. 변경 사항을 GitHub에 저장했습니다.

### 2.3 Step 3 — CLAUDE.md 설정하기

**2.3.1 프로젝트 규칙서, CLAUDE.md**
CLAUDE.md는 Claude Code가 프로젝트를 열 때 가장 먼저 읽는 파일입니다. 여기에 적어 둔 규칙은 매 대화마다 자동으로 적용됩니다. "파일 이름은 index.html로 고정", "한국어로 응답" 같은 규칙을 넣어 두면 매번 설명할 필요가 없습니다.

설정 방법

1. 아래 CLAUDE.md 내용을 전체 복사합니다.
2. Claude Code에 붙여넣으며 "이 내용으로 저장소 최상단에 CLAUDE.md 파일을 만들어 줘"라고 요청합니다.
3. Claude가 파일을 생성해 GitHub에 저장합니다.

파일 이름은 반드시 CLAUDE.md(대문자)이며, 위치는 저장소 최상단 폴더입니다.

CLAUDE.md (복사용)

```markdown
# CLAUDE.md

## 기본 규칙
- 파일 이름은 index.html로 고정합니다.
- 응답은 한국어로 합니다.
- 코드를 바꾼 뒤에는 무엇을 왜 바꿨는지 한 줄로 알려줍니다.

## 안전 규칙 (최소)
- 큰 변경을 하기 전에는 현재 동작하는 상태를 먼저 저장(커밋)해, 문제가 생기면 되돌릴 수 있도록 합니다.
- 변경 후에는 페이지가 정상 동작하는지 확인한 뒤 저장합니다.

## 2인 이상 팀으로 참가할 때
- GitHub로 협업할 경우 브랜치를 나눠 개발하고 PR로 합칩니다.
```

혼자(1인) 참가하는 경우 위 협업 규칙 없이 그대로 개발하면 됩니다. 팀원이 2명 이상이라면 3장(팀 협업)을 참고하세요.

### 2.4 Step 4 — Vercel로 배포하기

**2.4.1 GitHub로 로그인 후 프로젝트 가져오기**

1. vercel.com 접속 → Continue with GitHub
2. Add New → Project 클릭
3. 방금 만든 저장소 선택 → Import

**2.4.2 Framework Preset 설정**
Import 직후 설정 화면에서 Framework Preset을 선택합니다. 이 가이드처럼 index.html을 직접 만든 경우에는 **Other**를 선택하면 됩니다(별도 빌드 과정이 없기 때문입니다). Preset이 맞지 않으면 빌드가 실패할 수 있습니다.

**2.4.3 배포 후 링크 받기**

1. 설정을 확인한 뒤 Deploy 클릭
2. 1~2분 뒤 weather-project.vercel.app 같은 주소가 생성됩니다
3. 이 주소가 구동 URL(제출 항목 ①)입니다

참고: GitHub에서 index.html을 수정하면 Vercel이 자동으로 다시 배포하므로, 별도로 재배포할 필요가 없습니다.
배포 후 확인: 다른 컴퓨터에서도 링크가 열리는지, 슬라이더 등 인터랙티브 기능이 정상 작동하는지 확인하세요. 구동 환경은 크롬 브라우저 기준이며, 1366×768·1920×1080·2560×1440 해상도에서 정상 표시되어야 합니다.

### 2.5 Step 5 — 제출하기

**2.5.1 제출항목**
제출은 아래 4가지입니다. 어떤 환경에서 제작했는지 관계없이 동일하게 적용됩니다.

- **① 구동 가능한 URL**: 누구나 접속해 바로 조작할 수 있는 웹 주소입니다. Vercel, GitHub Pages, Netlify 등 어떤 배포 방식이든 인정됩니다. 로그인·결제 없이 열려야 하며, 구동 환경은 크롬 브라우저(해상도 1366×768·1920×1080·2560×1440 지원) 기준입니다.
- **② 소스 코드 (구글 폼 업로드)**: 완성된 소스를 압축(zip)해 운영팀이 안내하는 구글 폼에 업로드합니다. 만약 GitHub에서 코드를 다운로드 받는다면 "Download ZIP" 버튼을 클릭하여 압축된 zip 파일을 업로드 하세요. GitHub 링크 제출이 아니라 파일 업로드이며, node_modules·빌드 산출물은 제외하고 소스만 담으세요. (구글 폼 주소: 〈운영팀 별도 안내〉)
- **③ 주요 프롬프트 세션**: 각 팀의 프로젝트 수행 시 AI 활용 근거가 되는 주요 대화 세션을 .md 또는 .txt 포맷으로 내보내기 후, zip 파일로 압축하여 운영팀이 안내하는 구글폼에 업로드 합니다.
- **④ 구동 환경 / 배포 방법 문서 (README 등)**: 대회 이후 공공 교육을 위해 실행·배포할 수 있도록 필요한 사항을 꼼꼼하게 정리합니다. 사용한 외부 리소스(라이브러리·이미지·폰트·데이터)의 출처·라이선스도 함께 명시합니다.
- 실행에 필요한 모델과 데이터셋을 구글폼으로 제출한다. (필수)

**2.5.2 제출 전 체크리스트**

- URL이 크롬 브라우저에서 열리고 인터랙션이 정상 동작하는가
- 소스 zip을 구글 폼에 올렸는가 (node_modules·빌드 산출물 제외)
- 프로젝트에 사용한 대화 세션을 구글 폼에 올렸는가
- README(필수)를 작성했고, 실행/배포 방법과 외부 리소스 출처·라이선스가 적혀 있는가

**2.5.3 제출 유의사항**

- 제출한 URL은 시상식 일정(9월 셋째 주 예상)까지 유지되어야 한다.
- 8월 21일 오후 6시(논의 후 변경 가능) 이후 소스코드는 변경할 수 없으며, 심사 시 데모 테스트 화면 녹화로 진행될 예정이다.
- 검증 기간 중 실행 시 핵심 로직 수정 또는 소스코드 누락이 확인되면 탈락 처리된다.

기상청 API는 개발용 API key만 사용한다. 개발용 API key는 횟수 제한 없이 개발·테스트하기 위해 제공되며 8월 22일까지 유효하다. 단, 최종 제출 시 소스코드와 URL에는 반드시 본인 개인 계정으로 발급받은 API key를 사용해야 하며, 개발용 API key는 절대 제출하면 안 된다.

## 3. 팀 협업 (GitHub PR)

최종 소스 코드는 운영팀이 안내하는 구글 폼에 업로드해 제출합니다. 아래 GitHub PR 협업 방법은 팀(2인 이상)이 서로 방해 없이 함께 개발할 때를 위한 안내입니다. 1인 참가자는 이 장을 참고만 하면 되며, 혼자 작업할 때는 브랜치·PR 없이 개발한 뒤 완성된 소스 코드를 구글 폼에 올리면 됩니다.

팀원이 각자 다른 기능을 개발할 때는 브랜치를 나눠 작업하고 PR로 합칩니다. Claude Code에 요청하면 브랜치 생성부터 PR까지 대신 처리합니다.

### 3.1 팀원 초대하기

팀으로 참가한다면 가장 먼저 저장소에 팀원을 초대합니다. GitHub 저장소 → Settings → Collaborators에서 팀원의 GitHub 계정을 추가하면, 각자 같은 저장소에 연결해 함께 작업할 수 있습니다. 팀원이 저장소에 접근할 수 있어야 이후 브랜치·PR로 함께 작업할 수 있습니다.

### 3.2 브랜치 만들기

기능 개발을 시작할 때 main과 분리된 브랜치를 먼저 만듭니다. 각자 다른 브랜치에서 작업하면 서로 방해받지 않습니다.
프롬프트 예시: "새 브랜치를 만들어 줘. 브랜치 이름은 feature/wind-chart로 해 줘."
브랜치 이름 규칙: feature/기능명 형태가 일반적입니다. 예: feature/typhoon-map, feature/rainfall-slider, fix/api-error

### 3.3 기능 개발하기

브랜치 위에서 기능을 개발합니다. 평소처럼 Claude Code에 요청하면 됩니다.
프롬프트 예시: "바람 방향을 0~360도 슬라이더로 조절하면 화살표가 회전하는 인터랙티브 컴포넌트를 만들어 줘. index.html에 추가해 줘."

### 3.4 커밋하기

작업한 내용을 중간중간 저장(커밋)합니다. 커밋은 "이 시점까지의 스냅샷"이며, 문제가 생기면 이 시점으로 되돌아올 수 있습니다.
프롬프트 예시: "지금까지 변경한 내용을 커밋해 줘. 메시지는 '바람 방향 화살표 컴포넌트 추가'로 해 줘."
팁: 큰 작업이 끝날 때마다 커밋하세요. 메시지는 무엇을 했는지 한 줄로 적습니다. 커밋은 되돌리기·기록용이며, 최종 소스 제출은 구글 폼 업로드이므로 커밋 히스토리 자체는 심사 대상이 아닙니다.

### 3.5 PR 올리기

기능 개발이 끝나면 검토를 요청하는 PR을 올립니다. Claude Code가 제목과 설명까지 작성해 줍니다.
프롬프트 예시: "feature/wind-chart 브랜치를 main에 합치는 PR을 올려 줘. 제목은 '바람 방향 시각화 기능 추가'로 해 줘."

### 3.6 리뷰 요청하기

PR을 올린 뒤 팀원에게 리뷰를 요청합니다. Claude Code에 팀원 GitHub 아이디를 알려주면 리뷰어로 지정합니다.
프롬프트 예시: "이 PR에 OOO님 리뷰를 요청해 줘."
또는 GitHub 웹에서 직접 요청하기

1. GitHub에서 PR 페이지를 엽니다
2. 오른쪽 사이드바 Reviewers 옆 톱니바퀴 클릭
3. 드롭다운에서 리뷰를 요청할 팀원 선택
4. 선택 즉시 그 팀원에게 리뷰 요청 알림이 전송됩니다

### 3.7 리뷰 반영하기

팀원이 수정 요청(리뷰)을 남기면 그 내용을 Claude Code에 전달해 반영합니다. 리뷰 내용은 GitHub PR 페이지에서 확인한 뒤 그대로 복사해 Claude Code에 전달하면 됩니다.

### 3.8 머지(Merge)하기

리뷰가 완료되고 승인을 받으면 main 브랜치에 합칩니다. 머지 후 Vercel이 자동으로 최신 버전을 다시 배포합니다. 머지 후 배포 주소를 다시 열어 기능이 정상 동작하는지 확인하세요.

### 3.9 충돌(Conflict)이 생겼을 때
두 팀원이 같은 파일의 같은 부분을 동시에 수정하면 충돌이 생깁니다. 당황하지 말고 Claude Code에 상황을 그대로 전달하면 됩니다. 어느 쪽을 유지할지, 두 내용을 합칠지 물어보면 처리합니다.


팀 작업 권장 흐름: 기능별로 브랜치를 나눈다 → 각자 개발 후 PR → 팀원 1명 이상 리뷰 → 머지 → Vercel 자동 배포. main 브랜치에는 항상 동작하는 코드만 유지하는 것을 원칙으로 합니다.

## 4. 규정 (저작권·보안)

제출물은 저작권·라이선스 기준을 지켜야 합니다. 문제가 확인되면 규정에 따라 단계적으로 제재(소명 절차 포함)될 수 있으니 기준을 확인하세요.

### 4.1 저작권 가이드라인

제출물은 팀 간·외부 출처 대비 표절 검증과 제공 계정의 개발 세션 확인을 거칩니다. AI 사용 자체는 문제되지 않으나, 다른 팀이나 외부 출처의 코드를 그대로 사용하는 것은 허용되지 않습니다. 또한 AI로 코드를 생성했더라도 저작권·특허·보안에 대한 책임이 면제되지 않으며, 제출물에 대한 최종 책임은 참가자에게 있습니다.

| 항목 | 체크 사항 | 주의 필요 |
| --- | --- | --- |
| AI 생성 코드 | Claude Code와 같은 AI 서비스를 활용하여 생성한 코드 사용 가능 | 다른 팀과 동일한 코드 그대로 제출 금지 |
| 오픈소스 라이브러리 | MIT, Apache 2.0 등 상업적 허용 라이선스 | GPL 계열은 활용 범위에 제약이 있을 수 있음. README 명시 필요 |
| 이미지·아이콘 | CC0, Unsplash, 직접 제작 등 | 출처 불명 이미지, 유료 스톡 무단 사용 금지 |
| 폰트 | Google Fonts, Noto Sans 등 무료 폰트 등을 포함하여 추후 사용에 문제가 되지 않는 폰트 사용 | 상업용 폰트 무단 사용 금지. 라이선스 README 명시 |
| 데이터 | 데이터의 출처 및 활용 가능 범위 확인 | 원출처 표기 필수 |

README에 반드시 포함할 것: 사용한 외부 리소스 목록(라이브러리명·버전·라이선스), 이미지·폰트 출처, 데이터 출처 표기 등 필수적인 사항을 반드시 포함하여 표기.

### 4.2 API 키·비밀값 보안

API 키를 비롯한 모든 비밀값(키·토큰·인증 정보 등)은 코드에 노출되지 않도록 다뤄야 합니다. 기상청 API 키뿐 아니라 외부 서비스에서 발급받은 어떤 키라도 마찬가지입니다. GitHub에 올라간 코드와 배포된 웹페이지는 누구나 열어볼 수 있어, 노출된 키는 도용·과금 사고로 이어질 수 있습니다. API 키가 노출되면 실격 처리되므로 보안에 유의하여 설계하세요.

다음과 같은 노출에 유의하세요.

- API 키·토큰을 HTML·JS 코드에 직접 써넣는 경우
- 키를 GitHub 저장소에 커밋하는 경우
- 키를 Slack·PR·스크린샷 등에 노출하는 경우
- 노출된 키를 그대로 방치하는 경우

## 5. 자주 묻는 질문 (FAQ)

**5.1 코딩을 전혀 못하는데 가능한가요?**
가능합니다. 웹 Claude Code에 한국어로 요청하면 코드를 대신 작성합니다. 이 가이드의 순서대로 따라 하면 됩니다.

**5.2 팀원과 함께 작업하려면?**
GitHub 저장소 Settings → Collaborators에서 팀원을 추가하면 됩니다. 각자 Claude Code에서 같은 저장소에 연결해 작업할 수 있습니다. 최종 소스는 운영팀이 안내하는 구글 폼에 업로드해 제출합니다(GitHub 링크 제출이 아닙니다). 자세한 흐름은 3장 팀 협업을 참고하세요.

**5.3 문의는 어디에 하나요?**
문의 사항별 Slack 채널에 올려주세요. 운영팀이 직접 응대합니다. 정규 응대일은 매주 공지를 확인해 주세요.

**5.4 Claude Code 계정은 어떻게 받나요?**
본선 진출 15팀에 운영팀이 이메일로 초대합니다. 계정 접속에 문제가 있으면 Slack 채널에 남겨주세요.

**5.5 AI가 만든 코드를 그대로 제출해도 되나요?**
가능합니다. 이 대회는 AI 활용을 권장합니다. 다만 다른 팀이나 외부 출처의 코드를 그대로 복사하는 것은 표절로 처리됩니다. AI로 생성했더라도 우리 팀의 아이디어와 구성이 담겨 있어야 합니다.

**5.6 소스 코드는 어떻게 제출하나요?**
완성된 소스를 압축(zip)해 운영팀이 안내하는 구글 폼에 업로드합니다. GitHub 저장소 링크 제출이 아니며, node_modules·빌드 산출물은 제외하고 소스만 담으면 됩니다.

**5.8 터미널로 개발해도 되나요?**
됩니다. 이 가이드는 초보자용 웹 인터페이스를 기준으로 하지만, 터미널·IDE로 작업해도 무방합니다. 제출물 4가지 요건만 충족하면 됩니다.

**5.9 기획서에 안내된 '기상자료개방포털 대용량 API 서비스'는 반드시 사용해야 하나요?**
기획서상 안내는 참고 예시이며, 필수 조건은 아닙니다. 콘텐츠 제작에 필요하다면 공공데이터포털, 국가기상위성센터 등 다른 신뢰할 수 있는 공공 기상·기후 데이터도 함께 활용할 수 있습니다.

**5.10 대용량 API 서비스 카탈로그에 포함된 항목이라면 모두 활용 가능한가요?**
기상자료개방포털(기상청 데이터 허브)에서 제공 중인 데이터는 활용 가능합니다. 기상청 API를 활용해 콘텐츠를 개발하는 경우 개발에 필요한 대용량 API 키는 대회 측에서 제공합니다. 다만 카탈로그의 개별 항목이 팀의 콘텐츠에 실제로 적용 가능한지, 원하는 방식으로 활용될 수 있는지는 팀에서 직접 확인 후 판단해야 합니다.
